# 아키텍처 결정 기록 (ADR)

주요 설계 결정의 맥락, 대안, 근거를 기록. 6개월 후 본인이 "왜 이렇게 만들었지?"라고 물을 때 답을 찾기 위한 문서.

## 형식

각 ADR은 다음 4개 섹션으로 구성:

- **컨텍스트**: 어떤 상황에서 결정이 필요했나
- **결정**: 무엇을 선택했나
- **대안**: 고려했지만 안 고른 옵션과 이유
- **결과**: 이 결정의 장점, 단점, 트레이드오프

## ADR-001: LLM JSON 구조화 출력 + 코드 위임 매핑 + 단일 호출

**Status**: Active

### 컨텍스트

v1은 LLM에게 마크다운을 직접 생성시키는 방식이었다. 분야별로 4번 호출하고 종합 1회 추가, 총 5회. 각 분석 본문 안에 "전자신문에 따르면", "[기사 제목](URL)" 같은 출처와 링크를 LLM이 직접 적었다.

문제점:
- **출처 환각 위험**: LLM이 매체명·URL을 만들어내거나 잘못 매칭할 수 있음. 한 번이라도 발생하면 신뢰성 붕괴
- **호출 비용/속도**: 5회 호출은 토큰도 5배, 시간도 5배
- **분야 횡단 시각 부재**: 분야별 분석 단계에서 다른 분야 맥락을 모름. 종합 단계가 분석 4개의 메타 요약이 되어버림
- **마크다운 파싱 취약점**: 마크다운 형식이 살짝 깨지면(`**` 빼먹기, `##` 개수 틀리기) 후처리가 무너짐

### 결정

세 가지 결정을 동시에:

**1. LLM은 JSON으로만 응답**
- `response_mime_type="application/json"` (Gemini) 또는 `response_format={"type": "json_object"}` (OpenAI)로 강제
- 스키마는 시스템 프롬프트에 명시
- 마크다운 생성은 코드(Jinja2)가 담당

**2. 출처/링크는 코드가 자동 매핑**
- LLM은 `referenced_article_ids: [3, 7, 12]` 같은 인덱스만 반환
- 매체명은 `collectors.SOURCE_MAP` (도메인 → 매체명)으로 결정론적 매핑
- URL은 `Article.link` 그대로 사용
- LLM은 매체명·URL 한 글자도 만지지 않음

**3. 단일 호출로 전체 분석**
- 모든 분야 기사를 한 번에 LLM에 투입
- 메가 트렌드 + 분야 5개 분석을 1회 호출로 동시 생성

세 결정은 사실상 같은 철학이다: **LLM은 분석 데이터만, 형식과 결정론적 데이터는 코드**.

### 대안

**대안 1**: 마크다운 직접 생성 + 마커로 토글 표시 (Jinja2 + 파서)
- 장점: LLM 출력을 사람이 바로 읽을 수 있음, 직관적
- 단점: 마크다운 파싱 취약점, 출처 환각 위험, 슬롯 단위 정규식 파싱 필요
- 기각 이유: 출처 무결성 보장 어려움

**대안 2**: JSON 출력 + 슬롯별 LLM 호출 (메가 트렌드 호출, 분야별 호출 N개)
- 장점: 각 슬롯이 작아 max_tokens 부담 적음
- 단점: 비용·시간 폭증, 분야 횡단 시각 약화
- 기각 이유: 단일 호출의 장점 잃음

**대안 3**: LLM이 매체명 인라인 표기 강제 (시스템 프롬프트 규칙)
- 장점: 단순. 출처 표기가 본문에 자연스럽게 녹음
- 단점: 환각 1회면 신뢰성 회복 불가. 검증 코드 추가 필요
- 기각 이유: "신뢰는 코드가 보장, LLM은 분석에만 집중" 원칙

### 결과

**장점**
- **출처 표기(매체명·URL)의 환각 0%** — 코드가 ID→매체명/URL을 결정론적으로 매핑하므로 매체명을 지어내거나 URL을 조작하는 일은 구조적으로 불가능
- 단일 호출로 분야 횡단 시각 자연스럽게 확보
- JSON 파싱은 마크다운 파싱보다 압도적으로 견고
- 같은 JSON에서 Notion/이메일/슬랙 등 다양한 출력 채널 재사용 가능

**단점**
- JSON 출력이 max_tokens 도달로 잘리면 전체 실패 (실제 운영 중 한 번 발생, max_tokens=32000으로 해결 → [TROUBLESHOOTING L1](TROUBLESHOOTING.md#l1-llm-json-truncated))
- LLM이 결정론적 데이터(예: 매체명)를 본문에 인용하지 못함. 분석 본문이 약간 추상적인 느낌
- JSON 스키마 변경 시 코드 검증 로직도 같이 수정 필요
- **본문-출처 의미 일치는 보장하지 않음 (중요).** 코드 검증은 `referenced_article_ids`가 그 분야 입력 ID 집합(`valid_ids`)에 속하는지만 본다(analyzer.py). LLM이 본문엔 ID 3 내용을 쓰고 `referenced_article_ids: [7]`로 적어도, 7이 유효 범위면 통과한다. 즉 "삼성전자 양산"을 분석하고 SK하이닉스 기사 링크를 다는 미스매치가 가능하다. v1의 마크다운 직접 생성은 환각 시 매체명이 어색해 티가 났으나, 현 구조는 매체명·URL이 코드 생성이라 **미스매치가 그럴듯하게 위장된다(검증 불가능한 환각으로 이동).** "출처 환각 0%"는 *URL 표기*에 한정된 주장이며, *본문-출처 의미 일치*에는 적용되지 않는다.

**트레이드오프**  
출력 유연성을 일부 포기하고 신뢰성·견고함을 얻음. 매일 자동 운영하는 시스템엔 후자가 압도적 가치. 단, 위 "본문-출처 미검증"은 의식적으로 감수한 잔여 위험이다 — 저비용 안전망(예: referenced_article_ids 개수 ↔ 본문 길이 비례 임계, 의존성 0)은 운영 중 ID 혼동 빈도를 관찰한 뒤 도입을 판단한다. 의미 일치를 본격 검증하려면 추가 LLM 호출이나 한국어 NLP 의존성이 필요한데, ID 혼동의 추정 빈도(저빈도) 대비 과투자라 현 단계에선 보류.

**미해결 이슈**
- **insight 본문에 article_id가 raw 표기로 새어나오는 결함 (부분 해결, 2026-05-29).** 2026-05-27 운영서 확인: `(`[46]`)`, `(`[42], [43]`)` 형태로 ID가 본문에 노출됐고, 인사이트는 종합 통찰이라 그 ID가 해당 분야 입력에 없는 경우도 있어(정치 insight의 [46]=알루미늄) 매체명 치환 대상조차 불명확. 2단 안전망으로 대응: (1) 프롬프트 1차 — system_prompt 인사이트 작성 규칙에 "insight 본문에 article_id 표기 금지" 명시 추가. (2) 코드 2차 — `renderer._clean_insight_ids`가 잔여 표기를 정규식으로 제거 (`(\`[N]\`)` / `\`[N]\`` / `[N]` 세 패턴). 회귀 테스트 `tests/test_insight_id_strip.py`에 운영 케이스 + 변형 + 정상 케이스 박았다. 단 이는 insight에 한정된 부분 해결 — deep_issues·minor_issues의 본문-출처 의미 일치(referenced_article_ids가 본문 내용과 매칭되는가)는 여전히 미검증.
- deep_issues·minor_issues의 본문-출처 의미 일치 미검증. 측정 항목: deep_issue 1건당 referenced_article_ids 평균 개수, 본문 길이 대비 ID 비율의 이상치. 운영 중 ID 1개짜리 장문 분석이 잦으면 비례 임계 도입.

## ADR-002: Jinja2 외부 템플릿 분리

**Status**: Active

### 컨텍스트

JSON → 마크다운 변환이 필요해졌다. 변환 방식 선택:

1. 코드 안에 마크다운 문자열을 하드코딩 (f-string 등)
2. Jinja2 등 템플릿 엔진으로 외부 `.md.j2` 파일 분리
3. JSON에서 바로 Notion 블록 배열 생성 (마크다운 단계 생략)

운영 중 출력 형식 조정(섹션 추가, 이모지 변경, 헤더 깊이 등)이 잦을 것으로 예상됨.

### 결정

**Jinja2 + `templates/report.md.j2` 외부 파일.**

```
templates/
└── report.md.j2
```

### 대안

**대안 1**: 코드 안 하드코딩 (f-string)
- 장점: 의존성 없음
- 단점: 출력 형식 수정 시 코드 변경. 디자인 의도 파악 어려움
- 기각 이유: 운영 중 잦은 조정 예상

**대안 2**: JSON → Notion 블록 직접 생성 (마크다운 단계 생략)
- 장점: 한 단계 감소. 변환 오버헤드 없음
- 단점: 로컬 백업 .md 파일을 별도로 만들어야 함. Notion 외 출력 채널 추가 시 작업 중복
- 기각 이유: 마크다운이 중간 표현으로서 다른 채널에 재사용 가능

**대안 3**: 단순 문자열 치환 (`{{var}}` → str.replace)
- 장점: 의존성 없이 외부화
- 단점: 조건문/반복문 불가. minor_issues 같은 가변 리스트 처리 어려움
- 기각 이유: 표현력 부족

### 결과

**장점**
- 출력 형식 수정이 `.md.j2` 한 파일만 건드림. 코드 무수정
- 조건문(`{% if has_sufficient_data %}`), 반복문(`{% for issue in issues %}`) 자연스러움
- 로컬 백업 .md가 자동으로 생성됨 (Notion 변환 전 단계 산출물)
- 사람이 .md.j2를 봐도 출력 구조가 직관적으로 보임

**단점**
- 의존성 1개 추가 (Jinja2)
- 토글 같은 Notion 전용 요소는 마커(`▶ ###`) 우회 필요. Notion 파서에서 별도 해석
- **4단 변환(LLM JSON → Jinja2 → 마크다운 → Notion 블록 파서 → API)의 문자열 오염 방어가 부분적이고 일부는 미검증.** 단계별 현황:
  - *Notion 2000자 제한*: **방어됨.** notion_writer의 `NOTION_TEXT_CHUNK=1900` + `_split_long_text()`가 rich_text를 분할. (이전 AI 리뷰가 "분할 없다"고 했으나 코드에 존재함 — 다만 ADR에 기록이 없어 오해를 샀으므로 여기 명시.)
  - *Notion API 실패 시 산출물 보존*: **방어됨.** `upload_to_notion()`이 API 호출 *전에* 로컬 `.md` 백업을 저장하고, 실패 시 백업 경로를 반환한다. 즉 렌더링 이후 단계가 깨져도 그날 산출물은 `.md`로 남는다.
  - *마크다운 특수문자 — 실제로 깨짐 (재현 확인).* 한국 매체 제목엔 `[속보]`, `[단독]`, `[종합]`이 매우 흔하다. 템플릿(`report.md.j2:72`)이 출처를 `- [{{src.name}}] [{{src.title}}]({{src.url}})`로 출력하므로, 제목이 `[속보] 검찰…`이면 결과는 `[머니투데이] [[속보] 검찰…](url)` — 중첩 대괄호가 된다. notion_writer의 인라인 링크 정규식(`\[([^\]]+)\]\(([^)]+)\)`, notion_writer.py:47)은 링크 텍스트를 "닫는 대괄호가 아닌 문자"로만 보기 때문에 이 중첩 구조에서 `[text](url)` 매칭에 **실패한다**(2026-05-26 산출물의 서소문 붕괴 `[속보]` 제목으로 재현 확인). 실패 양상은 API 422 크래시가 아니라 **조용한 실패**다: 해당 줄 전체가 링크 없는 평문으로 출력되어 **출처 링크(매체명·제목·URL 전부)가 통째로 사라진다.** 크래시가 없어 운영 로그엔 정상으로 찍히고, 사용자가 그 줄 링크를 클릭하려 할 때만 발견된다 — silent dropping의 일종. `[속보]`/`[단독]`은 상시 등장하므로 저빈도가 아니라 **상시 발생**이다. (이전 판본은 이를 "미검증 위험"으로만 적었으나, 재현 결과 확정된 결함으로 정정.)
  - *JSON 백슬래시 깨짐(저빈도).* 분석 본문에 코드·정규식이 들어가면 이스케이프 누락으로 JSON 파싱 실패 → 그날 분석 0건. 뉴스 브리핑 특성상 코드 삽입 확률은 낮으나 0은 아니다. analyzer는 ``` 펜스 제거 후 재파싱하는 1차 폴백만 있고, 부분 복구는 없다.

**트레이드오프**  
의존성 작은 추가 비용으로 운영 편의성과 표현력을 얻는다. 정직한 비용 표기: 중간 표현 1단계로 디버깅 단순성과 채널 확장성을 얻는 대신, JSON 파싱 단계가 깨지면 그날 *분석*이 0건이 된다(단 렌더링 이후 단계 실패는 백업이 보존). 마크다운 특수문자 이스케이프는 미검증 영역으로 남아 있다.

**미해결 이슈**

## ADR-003: 하이브리드 본문 선별

**Status**: Active

### 컨텍스트

본문 크롤링은 매체별 봇 차단 등으로 100% 성공 보장 안 됨. 본문 확보 실패 시 처리 방법:

- **요약만이라도 분석 진행** (A): 분야 균형 유지, 단 환각 위험 ↑
- **본문 확보된 기사만 사용** (B): 분석 깊이 보장, 단 분야 균형 깨질 위험
- **하이브리드** (D): 본문 우선 + 부족 시 요약 보충 + 명시적 태깅

분석 깊이와 분야 망라성 둘 다 중요한 시스템.

### 결정

**하이브리드 (D안) + 명시적 태깅 + 재시도(E안 병행)**

`collectors.select_for_analysis()`:
- 본문 확보 기사 우선 정렬
- 분야당 본문 5건 이상이면 본문만 사용
- 5건 미만이면 본문 전부 + 요약 기사로 동적 한도(8~20건, ADR-014 참조)까지 보충
- LLM 입력 시 각 기사에 `[본문]` 또는 `[요약만]` 태그 명시

시스템 프롬프트에 강제:
> `[요약만]` 태그가 붙은 기사는 정보가 제한적이다. 해당 기사 기반 심층 분석은 자제하고, 본문이 있는 기사를 주축으로 분석한다.

### 대안

**대안 A (요약만이라도 진행)**
- 장점: 운영 안정성 최고, 데이터 손실 0
- 단점: 환각 위험 가장 큼. "어디서 본 듯한 일반론"으로 흐를 가능성
- 기각 이유: "전문가 분석"이라는 컨셉과 충돌

**대안 B (본문 기사만)**
- 장점: 분석 깊이 보장, 환각 위험 최저
- 단점: 정치/사회 분야가 0~3건만 남을 수 있음. 큰 사건도 본문 차단되면 누락
- 기각 이유: 매일 운영 시스템에서 분야별 분량 변동이 너무 큼

### 결과

**장점**
- 분석 깊이와 분야 균형 동시 확보
- LLM이 태그를 인지하면 환각이 줄어듦 (검증된 프롬프팅 패턴)
- 본문 부족 분야는 `limitation_note`로 솔직하게 표시되어 신뢰도 보전

**단점**
- 구현 복잡도 살짝 증가 (분기 로직 + 프롬프트)
- LLM이 태그를 무시할 가능성 (실측상 거의 안 일어남)

**트레이드오프**  
약간의 복잡도로 두 극단의 단점을 모두 피함. 검증된 프롬프팅 패턴이 환각 위험을 실용적 수준으로 낮춤.

## ADR-004: 3단계 본문 크롤링 재시도

**Status**: Active

### 컨텍스트

한국 매체는 봇 차단 정책이 매체마다 다르고 자주 변함. 단일 라이브러리(예: `trafilatura.fetch_url`)만으로는 본문 확보율이 50% 수준에 머무를 위험.

### 결정

3단계 폴백:

```
1단계: trafilatura.fetch_url + trafilatura.extract
       (기본 라이브러리 동작, 일부 매체에 효과적)
        ↓ 실패
2단계: requests(UA) + trafilatura.extract
       (UA 차단을 우회, HTML만 받아 trafilatura가 본문 추출)
        ↓ 실패
3단계: requests(UA) + BeautifulSoup 휴리스틱 셀렉터
       (article, div.article-body 등 한국 매체 공통 셀렉터)
```

성공 판정: 본문 길이 ≥ 300자.

### 대안

**대안 1**: 단일 라이브러리 (trafilatura만)
- 장점: 단순
- 단점: 봇 차단 매체에서 확보율 급락
- 기각 이유: 운영 첫날 80%+ 확보 안 되면 분석 품질 흔들림

**대안 2**: Selenium/Playwright (JS 렌더링 포함)
- 장점: JS로 본문 로드하는 매체도 처리 가능
- 단점: GitHub Actions에서 무거움. 매체당 5~10초 추가
- 기각 이유: 한국 매체 대부분 정적이라 효용 대비 비용 큼

**대안 3**: 매체별 전용 파서 (커스텀 셀렉터 매체별로 작성)
- 장점: 매체별 최적화
- 단점: 매체 추가/변경 시 코드 수정. 유지보수 부담
- 기각 이유: 일반 폴백으로 충분한 수준 달성

### 결과

**장점**
- 운영 첫 실행에서 본문 확보율 100% (335건 중 335건) 달성
- 매체별 분기 없이 일반화. 새 매체 추가 시 추가 작업 불필요

**단점**
- 실패 시 3번 시도 → 매체에 부담 + 시간 추가 (실패당 +5~10초)
- BeautifulSoup 휴리스틱은 매체 HTML 구조 변경 시 망가질 수 있음

**트레이드오프**  
실패 케이스의 시간 페널티는 dedup 단계에서 노이즈 줄여줘 큰 부담 아님. 운영 안정성이 보상함.

## ADR-005: 5분야 구성 + 매체 다양화

**Status**: Partially superseded by ADR-012

### 컨텍스트

v1은 4분야 (IT·테크·AI / 경제·금융·증시 / 정치·사회·시사 / 산업)였다. 평가에서 두 문제 지적:

- "정치·사회·시사"가 너무 광범위
- 한국경제 RSS가 3개 분야에 들어가 매체 편향 심함

### 결정

**5분야로 분리 + 진보/보수 매체 균형**

> ⚠️ **이 결정의 RSS→분야 매핑 부분은 [ADR-012](#adr-012-rss-평탄화--llm-사후-분야-분류)로 대체됨.**
> 한국 매체 RSS가 대부분 종합 피드로 판명되어 매체→분야 매핑이 불가능했고,
> 분류는 LLM이 사후 수행하는 구조로 전환됨.
> 다만 본 ADR의 **5분야 정의와 매체 다양성 원칙은 유효**하다.

초기 구성 (Dict 형식):

```python
RSS_FEEDS = {
    "IT·테크·AI": [IT조선, 전자신문 ×2],
    "경제·금융·증시": [머니투데이, 한국경제, 매일경제],
    "정치": [동아일보(보수), 경향신문(진보), 오마이뉴스(진보)],
    "사회·시사": [연합뉴스(중립), 동아일보(보수), 한겨레(진보)],
    "산업": [매일경제, 전자신문],
}
```

정치 분야는 종합지 위주(동아/경향/오마이)로 하고 한국경제(보수-경제지)는 제외. 이는 정치 보도에는 경제지 관점보다 종합지 관점이 적합하다는 판단.

### 대안

**대안 A**: 4분야 유지
- 장점: RSS 소스 적게 필요. 단순
- 단점: 정치·사회 합쳐서 분석하면 두 분야 중요 이슈가 서로 묻힘
- 기각 이유: 정치와 사회는 분석 결의 톤이 달라야 함

**대안 B**: 3분야로 통합 (산업을 경제에 합침)
- 장점: 각 섹션이 더 두꺼워짐. 분석 분량 보장
- 단점: 산업 흐름이 경제 섹션에 묻힘. 시각 구분 어려움
- 기각 이유: 산업과 경제는 분석 시각이 다름

### 결과

**장점**
- 정치 분야가 종합지 3개로 이념 분산
- 사회·시사가 연합(중립)·동아(보수)·한겨레(진보)로 균형
- 분야별 색깔이 명확해짐

**단점**
- 산업 분야는 매체 2개뿐 (대체 매체 부족)
- 정치 분야가 진보 2 + 보수 1 구성. 동아 50건이 합산보다 많을 가능성도 있어 실제 비율은 미지수
- RSS 매체 변경/폐지 시 분야 0건 발생 가능 → [TROUBLESHOOTING R1](TROUBLESHOOTING.md#r1-rss-url-dead)

**트레이드오프**  
완벽한 이념 균형은 어렵지만 의식적 분산만으로도 v1의 한경 편향은 해소됨.

## ADR-006: 한 페이지 통합 + 토글 활용

**Status**: Active

### 컨텍스트

매일 보고서를 어떻게 나눌지 결정:

- **A**: 한 페이지에 5분야 다 통합
- **B**: 인덱스 페이지 + 분야별 페이지 5개 (총 6 페이지/일)
- **C**: 이슈별 페이지 분리 (총 20~30 페이지/일)

본인이 우려한 점: 한 페이지에 다 담으면 너무 길어진다.

### 결정

**A안 (한 페이지 통합) + 토글 헤더 적극 활용**

페이지 구조:
- 제목 + TL;DR 콜아웃
- 메가 트렌드
- 분야 5개 (각 분야 안에 deep_issues 토글, minor_issues 토글)
- 푸터

길이 문제는 토글 헤더(`is_toggleable=True`)로 해소:
- deep_issues 각 이슈가 토글 헤더 (기본 접힘)
- minor_issues 전체가 하나의 토글 (기본 접힘, `(N건)` 표시)

접힘 상태 페이지 길이: 약 80~90줄. 펼침 상태 최대 280줄.

### 대안

**대안 B (분야별 페이지 분리)**
- 장점: 분야별 추적 자산화. 페이지당 길이 짧음
- 단점: 메가 트렌드는 어디에 두지? 인덱스 페이지 별도 필요 → 사실상 한 페이지 + 5개 추가 = 6개. API 호출 6배. 모바일에서 클릭 부담
- 기각 이유: 분야 횡단 시각이 중심 가치인데 분리하면 약화

**대안 C (이슈별 페이지)**
- 장점: 이슈 단위 추적 가능
- 단점: 하루 20~30개 페이지 생성. 매일 통독 부담이 옵션 A의 정반대 극단
- 기각 이유: 매일 통독 모델과 충돌

### 결과

**장점**
- 매일 5분 통독 가능한 분량
- 분야 횡단 시각(메가 트렌드)이 자연스러움
- Notion 사이드바 목차에서 분야/이슈로 즉시 점프
- API 호출 1회 페이지 1개 → 단순

**단점**
- 한 페이지가 길어 분야별 동시 추적은 약함
- 1~2개월 누적 후 "지난주 IT만" 같은 검색은 캘린더뷰 + 본문 키워드 검색에 의존

**트레이드오프**  
MVP로 적합한 단순함. 운영하다 분야별 추적 필요성이 강해지면 B안 전환 가능 (기존 자료 자산화 안 깨짐).

## ADR-007: Notion DB row 방식

**Status**: Active

### 컨텍스트

매일 페이지를 어떻게 저장할지:

- 부모 페이지 하위에 페이지를 단순히 흩뿌리기
- Notion Database의 row(=페이지)로 추가

### 결정

**Notion Database + row 방식.** DB 속성(날짜, 모델, 본문 확보율, 분야 태그, 분석 기사 수)으로 메타데이터 구조화.

### 대안

**대안 1**: 단순 페이지 흩뿌리기
- 장점: 가장 단순. DB 초기 설정 불필요
- 단점: 1~2개월 후 부모 페이지가 스크롤 지옥. 검색·필터 불가
- 기각 이유: 누적 자산화 가치 잃음

**대안 2**: 데이터베이스 템플릿 기능 활용
- 장점: 시각 디자인 자유로움
- 단점: 템플릿은 UI에서 "템플릿 적용" 클릭해야 채워짐. API로 row 추가 시 자동 적용 안 됨
- 기각 이유: 자동화 흐름과 충돌

### 결과

**장점**
- 캘린더뷰, 필터, 정렬, 그룹화 모두 사용 가능
- 본문 확보율 같은 운영 지표를 Notion에서 직접 정렬·차트화 가능
- 모델 사용 분포(gemini-2.5-flash vs gpt-5-mini)로 폴백 빈도 추적
- 분야 태그로 "특정 분야 데이터 있던 날" 빠른 필터

**단점**
- 2025-09-03 API 변경으로 data_source 구조 학습 곡선 발생 (실제 운영 중 직면 → [TROUBLESHOOTING N6](TROUBLESHOOTING.md#n6-notion-api-2025-09-03))
- DB 초기 설정에 추가 단계(`init_db.py` + 속성 수동 보완) 필요 → [TROUBLESHOOTING N1~N5](TROUBLESHOOTING.md#notion-api설정)

**트레이드오프**  
초기 설정의 진입 장벽 약간 추가, 운영 가시성과 자산화는 큰 이득.

## ADR-008: minor_issues 가이드라인만

**Status**: Active

### 컨텍스트

deep_issues 외에 분야별 "기타 이슈"도 다루기로 결정 후, 개수 상한을 둘지 논의:

- 상한 둠 (예: 최대 5개)
- 상한 없음 + 가이드라인

### 결정

**상한 없음 + 가이드라인 시그널**

초기 시스템 프롬프트:
> minor_issues는 그날 분야에서 실제로 발생한 추가 이슈를 빠짐없이 짚으세요. 단, 광고성·인사·단순 동향 변동(주가 등락 등 분석 가치 낮은 사실 보고)은 제외합니다. 의미 있는 추가 이슈가 없으면 빈 배열로 두세요.

토글 라벨에 `(N건)` 표시로 개수 가시화.

> **(2026-06-03 개정)** 위 "빠짐없이 짚으세요"가 광고성 제외보다 강하게 작동해
> 칸을 채우는 부작용이 실측됨(아래 결과 참조). 가이드라인을 "빠짐없이"에서
> **기능 기반 포함 테스트**로 교체. 상한 없음 원칙(하드 상한 안 둠)은 유지하되,
> 포함 자격을 명시. 상세 문구는 analyzer.py [minor 포함 테스트] 참조.

### 대안

**대안 1**: 상한 둠 (최대 5개)
- 장점: 페이지 길이 안전망. 토큰 제어
- 단점: 상한 수치가 임의적. 분야에 큰 사건 많은 날도 5개로 제한
- 기각 이유: 데이터에 충실하지 않음. 토큰은 무료 한도 내라 페이지 길이도 토글로 해소됨

**대안 2**: 기사 수의 절반 이내
- 장점: 데이터 비례
- 단점: LLM이 비율 정확히 따를지 불확실
- 기각 이유: 가이드라인의 한 종류일 뿐, 자유 판단과 큰 차이 없음

### 결과

**장점**
- 분량 변동이 자연스러움. "오늘은 minor 8건" / "오늘은 0건" 자체가 신호
- 운영 중 시스템에 손 안 대고 데이터 그대로 반영
- 토글로 묶이므로 페이지 길이엔 영향 없음

**단점**
- ~~LLM이 망라성 강박으로 의미 없는 이슈까지 만들 가능성 (실측상 거의 안 일어남, 가이드라인이 효과적)~~ **(2026-06-03 정정: 실측됨.)** 06-02 산출물에서 산업 minor 5건 전부가 광고성(신제품·할인·분양: 더벤티 레드불 협업, 세븐일레븐 상생, 롯데마트 무가당, 신세계 카드위크, 천안 분양 르포)으로 채워짐. 경제에도 도서 홍보("염승환 ETF") 1건. 원인은 "빠짐없이 짚되" 지시가 광고성 제외보다 강하게 작동 + 산업 풀에 소비재 신제품 기사 비율이 높은 날이 겹친 것. (AI 리뷰는 ADR-014 선별 확대를 원인으로 지목했으나 **오진** — 산업은 SELECT_MIN=8에 묶인 분야라 ADR-014 동적 할당과 무관하게 늘 8건이었다. 진짜 원인은 ADR-008의 "빠짐없이" 가이드라인이다.)

**대응 (2026-06-03)**
"빠짐없이"를 **기능 기반 포함 테스트**로 교체(analyzer.py). 핵심 3요소:
1. *확장성 + 지속성 테스트*: 개별 기업 활동을 넘어 산업 구조·추세·정책으로 연결되는가(확장성) + 1주일 후에도 언급할 만한가(지속성). 둘 다 만족해야 포함. 형식(신제품/할인)이 아니라 기능(구조 연결 여부)으로 판별 — 같은 기업이라도 "신세계 카드위크"(제외)와 "신세계 초고가 주얼리 시장 공략=소비 양극화"(포함, 05-30 실제 deep)가 갈린다.
2. *출처 강제*: "추세의 일부"라고 격상하려면 그 추세를 보여주는 다른 기사가 풀에 1건 이상 있어야 함. 단일 기사로만 존재하면 구조적 추세로 격상 금지. LLM의 자의적 "이것도 트렌드" 우기기를 풀의 객관적 사실로 묶는다. (단 LLM이 풀 대조를 환각할 수 있어 완벽한 게이트는 아님 — 강한 휴리스틱.)
3. *반대 신호*: "0~2건이 정상, 0건도 정상 신호, 5건 이상이면 느슨함의 신호니 다시 거르라." 하드 상한(대안 1에서 기각)이 아니라 소프트 자기점검이라 상한 없음 원칙 유지. 숫자 5는 06-02 산업 5건 관측 기반.

회귀 테스트 불가(LLM 출력)→운영 검증. 다음 산출물에서 광고성 minor 비율이 핵심 검증 포인트.

**트레이드오프**  
LLM의 판단 능력을 신뢰하는 선택. 단 "빠짐없이"가 신뢰를 잘못된 방향(망라성)으로 밀었음이 드러나, 신뢰의 방향을 "기능 판별"로 재설정. 상한 없음 철학은 유지.

## ADR-009: Gemini Flash + OpenAI 폴백

**Status**: Active

### 컨텍스트

v1은 Claude Opus 4.7 사용. 비용 추정 결과 월 8만원 수준. 사용자가 비용 지양 의향 표명.

### 결정

**1차: Gemini 2.5 Flash (무료 티어), 폴백: GPT-5 mini**

선택 흐름:
1. Claude Opus 4.7 (월 8만원, 너무 비쌈)
2. Claude Haiku 4.5 (월 ~4천원, 합리적이지만 Gemini 무료가 가능)
3. Gemini 2.5 Flash 무료 (선택, 일일 한도 내)
4. 폴백 LLM 후보: Claude Haiku vs OpenAI GPT-5 mini
5. GPT-5 mini 채택 (Claude Haiku 대비 저렴, 한국어 품질 비등)

### 대안

**대안 1**: Claude Haiku 4.5 단독
- 장점: 한국어 품질 안정. 단일 모델로 단순
- 단점: 월 ~4천원 발생. 무료 옵션 있는데 굳이
- 기각 이유: 비용 지양 의향

**대안 2**: Gemini 단독 (폴백 없음)
- 장점: 비용 완전 0원. 가장 단순
- 단점: Gemini 503 발생 시 그날 분석 못 함 (실제 발생함)
- 기각 이유: 매일 운영 시스템엔 100% 가용성 보장 어려움

**대안 3**: 로컬 LLM (Ollama)
- 장점: 비용 0원
- 단점: GitHub Actions runner에서 못 돌림. 자체 PC 24시간 가동 필요
- 기각 이유: 인프라 복잡도 폭증

### 결과

**장점**
- 평시 비용 0원
- 폴백 작동 시도 GPT-5 mini는 월 5회 가정 시 약 200원
- 한국어 분석 품질 두 모델 모두 양호

**단점**
- Gemini 무료 티어 안정성 보장 없음 (실측: 운영 초기 503 1회, 2026-06-01 분석 단계 503으로 그날 빈 브리핑 생성 → [TROUBLESHOOTING L2](TROUBLESHOOTING.md#l2-gemini-overload))
- 운영 중 OpenAI 결제 등록은 추가 단계

**재시도 vs 폴백 — 다른 계층 (2026-06-01 추가)**  
2026-06-01 503 장애로 드러난 설계 갭: **폴백만 있고 재시도가 없었다.** 둘은 다른 계층이다.
- *재시도*: 일시적 과부하(503 UNAVAILABLE, 500 INTERNAL) 대응. 같은 모델이 몇 초~분 뒤 복구되므로 backoff 후 재시도하면 대부분 풀린다.
- *폴백*: 1차 모델의 지속적 장애 대응. 다른 모델로 갈아탐(비용 발생).

503에 폴백부터 태우는 건 잘못 — 잠시 후 풀릴 Gemini를 두고 유료 OpenAI로 가는 낭비. **올바른 순서: 재시도(3회, 4→8→16초 지수 backoff) → 다 실패하면 폴백 → 폴백도 없으면 raise.** `GeminiProvider.generate`가 `tenacity.Retrying`으로 `generate_content` 호출을 감싼다(llm.py). 재시도 대상은 **ServerError(5xx)만** — ClientError(4xx: 400/401/403/429)는 재시도해도 같은 실패라 즉시 통과시킨다. 특히 429(한도 초과)를 재시도로 뚫으려 하면 안 됨. 회귀 테스트 `tests/test_llm_retry.py`에 503 재시도/4xx 즉시실패/429 비재시도/복구 시나리오를 박았다.

**미해결 이슈**
- OpenAI 폴백 키 미설정 상태 (2026-06-01 기준). 재시도로 일시적 503은 막지만, Gemini 지속 장애 시엔 여전히 무방비. 키 등록은 본인 결정 대기.
- 재시도 3회(최대 ~28초)가 GitHub Actions 분석 단계 타임아웃 안에서 충분한지는 장기 장애 발생 시 관찰 필요.

**트레이드오프**  
운영 비용을 0~수백원 수준으로 압축. 일시적 과부하는 재시도로, 지속 장애는 폴백으로 대응하되 폴백 활성화(OpenAI 키)는 본인 결정에 맡김.

## ADR-010: 로컬 환경변수는 export만

**Status**: Active

### 컨텍스트

API 키 관리 방법:

- 셸 `export` 명령
- `.env` 파일 + `python-dotenv`
- `~/.zshrc` 등 셸 rc 파일에 영구 export

### 결정

**`export` 방식 (옵션 A)**

코드는 `os.environ.get()`만 사용. `python-dotenv` 미설치.

### 대안

**대안 1**: `.env` 파일 지원
- 장점: 매번 export 안 해도 됨
- 단점: 의존성 1개 추가. `.gitignore` 빠뜨리면 키 유출 사고
- 기각 이유: 로컬 실행은 `init_db.py` 1회 정도. 빈도 낮아 export로 충분

**대안 2**: 셸 rc 파일 영구 export
- 장점: 터미널 켤 때 자동 적용
- 단점: 시스템 전체에서 키 노출 (다른 프로젝트도 사용 가능)
- 기각 이유: 보안 측면에서 부적절

### 결과

**장점**
- 의존성 추가 없음. 단순함
- 키 유출 위험 최저 (디스크에 저장 안 됨)
- GitHub Actions는 Secrets로 별도 관리, 로컬과 분리

**단점**
- 로컬 반복 실행 시 매번 export 필요
- 다른 사람과 협업 시 키 공유 어려움

**트레이드오프**  
운영은 GitHub Actions가 하니까 로컬 실행 빈도가 낮음. 빈도 낮으면 export로 충분.

**트레이드오프**  
개인 운영 도구 규모에 맞춘 실용적 선택. 문서가 10개를 넘으면 그때 `docs/` 폴더 전환 검토.

## ADR-012: RSS 평탄화 + LLM 사후 분야 분류

**Status**: Active

### 컨텍스트

운영 1회차 결과를 AI 평가에 부친 결과, **분야 분류 누수가 가장 큰 문제**로 드러남:
- IT·테크·AI 분야 deep_issues에 스타벅스 논란, 삼성 노조, 미-이란 우라늄 등 IT 무관 기사 다수
- 한 사건(스타벅스)이 4개 분야에 동시 등장

원인 분석 (`rss_probe.py` 샘플 제목 검증):
- **IT조선 allArticle.xml**: "전체" 피드. 정치/사회/스포츠 다 섞임
- **전자신문 902/901/904**: 모두 종합 피드 (IT 메인 섹션이 아님)
- **머니투데이 mt_news.xml**: 사실상 연예·스포츠 피드
- **매경 30000001/40300001**: 종합 피드
- **정치/사회만** 매체별 섹션 RSS 깨끗 (동아 politics.xml 등)

즉 **한국 매체의 RSS는 대부분 종합 피드**. ADR-005의 매체→분야 매핑은 현실에서 작동 불가.

### 결정

세 가지 변경을 동시에:

**1. RSS_FEEDS 자료구조 평탄화**
- `Dict[str, List[str]]` (분야별 매체 매핑) → `List[str]` (단일 매체 풀)
- 모든 매체에서 한 풀로 수집

**2. Article.category를 Optional로**
- 수집 시점엔 None
- LLM 분류 후 채워짐
- `is_cross_category: bool` 필드 추가 (분야 경계 사건 표시)

**3. LLM 분류 단계 신설 (classifier.py)**
- 제목만 입력 (본문 안 봄, 토큰 절약)
- 6개 분야 코드 중 하나 + cross 플래그 반환
- 코드가 매핑하여 분야명으로 변환
- "기타"는 분석에서 제외

### 대안

**대안 1**: 다른 매체의 섹션 RSS 추가 탐색
- 장점: 기존 구조 유지
- 단점: 한국 매체 대부분 섹션 RSS 없음. 또 죽거나 누수일 가능성
- 기각 이유: 근본 해결 아님

**대안 2**: 키워드 기반 분류 (코드만, LLM 안 씀)
- 장점: 호출 비용 0, 빠름
- 단점: "삼성전자"가 IT인지 산업인지 등 모호한 경우 처리 불가. 분야 정의가 1:N
- 기각 이유: 분류는 사실이 아닌 판단(judgment). LLM이 적합

**대안 3**: 분류와 분석을 LLM 1회 호출에 통합
- 장점: 호출 1회 유지
- 단점: 프롬프트 복잡도 ↑, JSON 스키마 복잡, LLM 부담
- 기각 이유: 분류와 분석은 책임이 달라 분리가 명확. 폴백 처리도 별개

### 결과

**장점**
- 분야 분류 정확도 95%+ (운영 검증)
- 한 사건이 여러 분야 등장 문제 거의 해결 (스타벅스가 사회 분야에만 집중)
- 매체 추가/제거가 단순 (`RSS_FEEDS.append()`)
- ADR-001의 "LLM은 판단, 코드는 사실" 원칙 보존: 분야는 판단(1:N), VALID_CATEGORIES 집합으로 검증

**단점**
- LLM 호출 1회 추가 (분류 + 분석 = 2회)
- 분류 LLM 실패 시 모든 기사 "기타"로 폴백 → 그날 분석 산출 안 됨
- 분류 정확도가 LLM에 종속 (경계 사례: "방송 접근권 TV"가 IT vs 사회)

**트레이드오프**  
호출 1회 추가 비용으로 분류 누수라는 근본 문제 해결. 무료 한도 충분히 여유.

**관련 결정**: [ADR-013 분류 LLM 토큰 최적화](#adr-013-분류-llm-토큰-최적화), [ADR-001 LLM 판단 vs 코드 사실](#adr-001-llm-json-구조화-출력--코드-위임-매핑--단일-호출) (분류는 판단에 해당)

## ADR-013: 분류 LLM 토큰 최적화

**Status**: Active

### 컨텍스트

ADR-012 도입 후 분류 LLM이 반복 잘림(JSON 파싱 실패). 진단 데이터로 원인 추적:

| 시도 | max_tokens | 결과 | 진단 |
|---|---|---|---|
| 1차 | 8000 | char 792에서 잘림 | 8000 부족 |
| 2차 | 32000 | char 11938에서 잘림 | 한국어 분야명 비효율 의심 |
| 3차 | 32000 + 단축 코드 | char 17530에서 잘림 | 여전히 잘림. 다른 원인 |
| 4차 (진단) | 출력 메타데이터 확인 | `output=433, total=12023` | **차이 7500 토큰이 thinking** |

**Gemini 2.5 Flash는 thinking이 기본 활성화**. thinking 토큰이 `max_output_tokens` 한도를 잠식. 응답에 사용 가능한 토큰이 극히 적음.

### 결정

세 가지 최적화 동시 적용:

**1. 분야명 단축 코드**
- LLM 응답에서 분야명을 한국어 그대로 받으면 토큰 효율 매우 나쁨 ("IT·테크·AI" = 약 10 토큰)
- 단축 코드 사용: `IT`, `ECO`, `POL`, `SOC`, `IND`, `MISC`
- 코드가 `CODE_TO_CATEGORY` 매핑으로 분야명 복원
- 객체당 출력 약 56% 절감

**2. 100건 배치 분할**
- 한 번에 전체 분류 시 출력 너무 길어 잘림
- `CLASSIFY_BATCH_SIZE = 100`
- 280건이면 3회 호출
- 호출별로 `offset`으로 ID 정렬 유지

**3. `thinking_budget=0` 명시**
- `llm.py`의 `generate()`에 `thinking_budget: Optional[int] = None` 파라미터 추가
- 분류 호출 시 `thinking_budget=0` 전달 (thinking 비활성화)
- 분석 호출은 미전달 (기본값 None → thinking 활성화 유지)

### 대안

**대안 1**: max_tokens를 더 크게 (64000)
- 장점: 단순. 코드 변경 1줄
- 단점: thinking 토큰이 출력 한도 잠식하는 본질적 문제는 그대로. 64000 한도로도 thinking이 50000+ 쓸 가능성
- 기각 이유: 진단 데이터로 thinking이 원인임이 확정됨

**대안 2**: 분류를 Gemini 2.0 Flash 등 thinking 없는 구버전 모델 사용
- 장점: 코드 변경 적음
- 단점: 모델 분기 관리 부담. 같은 코드 경로에서 모델 다름
- 기각 이유: `thinking_budget=0`이 더 단순한 해결

**대안 3**: 분류를 GPT-5 mini로 (Gemini 아예 안 씀)
- 장점: OpenAI는 thinking 잠식 없음
- 단점: 결제 등록 필요, 폴백이 아닌 1차로 사용 시 비용 발생
- 기각 이유: 무료 한도 유지가 우선

### 결과

**장점**
- 분류 응답이 안정적으로 끝까지 완료 (`finish_reason=STOP`, `output ≈ 2000~3000 토큰`)
- 분석 호출은 thinking 유지로 분석 품질 보존
- `thinking_budget` 파라미터 신설로 향후 호출별 제어 가능

**단점**
- 분류 호출 횟수 3배 (배치 분할)
- 단축 코드 매핑 코드 추가 (`CODE_TO_CATEGORY`)
- thinking 비활성화로 분류 품질이 약해질 가능성. **이 결정의 실제 동기를 정직하게 밝힌다: thinking을 끈 것은 품질 판단이 아니라 출력 안정성 때문에 불가피했다.** 위 진단표대로 thinking이 `max_output_tokens`를 잠식해 JSON이 매번 잘렸고(`output=433/total=12023`), thinking을 끄지 않으면 분류가 **작동 자체를 못 했다.** 즉 "분류엔 thinking이 필요 없어서 껐다"가 아니라 "thinking을 켜면 분류가 불가능해서 껐고, 그 대가로 분류 품질을 일부 희생했다"가 정확하다.
  - AI 리뷰(2-②)는 "분류가 판단이면 thinking이 필요하고, 패턴 매칭이면 LLM 쓸 이유가 약하다. 둘 중 하나는 거짓"이라 했다. 이 이분법 자체는 거짓이다(판단에도 난도 스펙트럼이 있어, 단일 분류 판단은 다단계 추론보다 thinking 한계효용이 낮다 — 분석 단계엔 thinking을 살려둔 이유). 그러나 비판이 **옳게 찌른 지점**은 따로 있다: 원본 ADR이 "분류는 판단이라 LLM이 적합"(ADR-012)이라 해놓고 그 판단력의 핵심 보조 수단(thinking)을 끈 모순을, 마치 품질상 무해한 선택인 양("패턴 매칭이라 영향 없음") 포장한 것. 실제로는 출력 제약에 떠밀린 트레이드오프였다. 이전 판본의 "추론보다 패턴 매칭", 그리고 그 직전 정정의 "저난도라 한계효용 낮음" 둘 다 **결정 동기를 사후 정당화한 수사였고 정정한다.**
  - 잔여 위험: thinking 없는 분류의 정확도 저하분은 미측정. 운영상 큰 오분류는 안 보였으나(2회 운영 95%+), thinking을 켰을 때와의 정확도 차이는 출력 잘림 때문에 비교 불가능하다. thinking을 켜고도 출력이 안 잘리게 하려면 배치 크기를 100→30 수준으로 더 잘게 쪼개 thinking 예산을 확보하는 방법이 있으나, 호출 횟수가 3회→9회로 늘어 무료 한도(분당 10회)를 압박한다. 정확도 저하가 운영 중 실제 문제로 드러나면 이 방향을 검토.

**트레이드오프**  
호출 3배 = 분류 단계 1초 → 3초. 무료 한도엔 무영향. 안정성 확보가 절대적으로 우위.

**관련 결정**: [ADR-012 LLM 사후 분류](#adr-012-rss-평탄화--llm-사후-분야-분류), [ADR-009 Gemini Flash 선택](#adr-009-gemini-flash--openai-폴백)

## ADR-014: 분야별 분석 한도 동적 할당

**Status**: Active

### 컨텍스트

ADR-003에서 `select_for_analysis`의 `max_total=8` 고정으로 정함. 운영 결과:

- 정치 95건 풀에서 8건 선별 → **87건이 사실상 버려짐** (8%만 활용)
- IT 33건 풀에서 8건 선별 → 24% 활용
- 분야별 풀 크기가 크게 달라 동일 상한이 비합리적

또한 급박한 특보 상황(예: 대형 사고 → 정치 200건+ 폭주)에 대비 부족.

토큰 측면에선 여유 충분:
- 평시 분석 입력 약 32k 토큰 (Gemini 1M 한도의 3%)
- max_total 늘려도 한도 영향 없음

### 결정

**풀 크기 비례 동적 할당** (main.py에서 분야별로 계산):

```python
SELECT_RATIO = 0.15      # 풀의 15%
SELECT_MIN = 8           # 최소
SELECT_MAX = 20          # 상한 (급박 특보 안전망)

target = max(SELECT_MIN, min(SELECT_MAX, int(len(deduped) * SELECT_RATIO)))
selected = select_for_analysis(deduped, max_total=target)
```

동작 예시 (오늘 데이터):
- 정치 94건 → 14건
- 사회 63건 → 9건
- 산업 47건 → 8건 (MIN 적용)
- IT 32건 → 8건 (MIN 적용)
- 경제 22건 → 8건 (MIN 적용)

급박 상황:
- 정치 150건 → 20건 (MAX 적용, 안전망)

### 대안

**대안 1**: 고정 max_total=12 (AI 평가 권장)
- 장점: 단순, 예측 가능
- 단점: 큰 분야 활용도 여전히 낮음. 급박 상황 무대응
- 기각 이유: 동적 처리가 더 데이터 비례적

**대안 2**: 본문 확보 기사 수 기반
- 장점: 분석 품질 직결
- 단점: 본문 확보율 100%인 현재 풀 크기 비례와 동일
- 기각 이유: 미래에 본문 확보율 떨어지면 가치 발생. 지금은 의미 없음

**대안 3**: 토큰 예산 균등 분배 (분야 5개 × N 토큰)
- 장점: 토큰 예산 명확
- 단점: 풀 크기 무시. 정치 95건이든 22건이든 동일 분배
- 기각 이유: 동적 할당의 의미가 약함

**대안 4 (보류)**: 임베딩 기반 dedup으로 풀 크기 자체 축소
- 장점: 근본 해결. 같은 사건 클러스터링
- 단점: `sentence-transformers` 의존성 무거움 (torch ~800MB), GitHub Actions 시간 증가
- 보류 이유: 운영 안정화가 우선. 1~2주 운영 후 dedup 약점 데이터 보고 도입 검토

### 결과

**장점**
- 큰 분야 활용도 향상 (정치 8% → 15%)
- 급박 상황에 자동 확장 (max 20까지)
- 토큰 컨텍스트 한도는 여유 (분석 입력 32k → 48k, Gemini 컨텍스트 1M의 5%). **단 이는 *컨텍스트 한도*이지 *효과적 어텐션 한도*가 아니다** — 모델 카드의 1M과 실제 정보 활용 가능 길이는 다른 개념이며, 긴 입력 중간 기사를 모델이 사실상 무시하는 Lost-in-the-Middle 위험은 별개로 남는다(아래 미해결 이슈 참조).
- main.py 한 곳에서 파라미터 관리

**단점**
- `ratio=0.15`는 휴리스틱. 이론적 근거 약함 (LLM 분석 효율 정점이 12~15건이라는 가정)
- 풀 크기와 분석 품질의 비례성 불완전 (큰 풀에 중요 사건이 더 많다는 보장 없음)
- **MIN/MAX·ratio를 5분야에 일률 적용.** 정치는 한 사건에 5~10건이 따라붙는 클러스터형 풀, IT는 비교적 독립 사건형 풀이라 특성이 다른데 같은 비율·천장을 쓴다. "동적"이라는 이름에 비해 분야 특성을 반영하지 못한다. 진짜 동적은 분야별 *사건 클러스터 수*에 비례해야 하나, 클러스터 수 산정엔 임베딩/clustering 의존성이 필요해 보류(아래 미해결 이슈).
- 운영 데이터 1~2주 후 재조정 필요

**측정 계획 (재조정 트리거)**  
"1~2주 후 재조정"을 구체화한다. 다음을 운영 로그에서 **먼저 관찰**한다 (지표만 정하고, 대응 임계 숫자는 아직 정하지 않는다 — 이유는 항목 끝에):
- **분야별 dedup 후 풀 크기 분포**: MAX=20 천장에 닿는 분야가 며칠에 한 번 발생하는지 빈도를 기록. (천장 통제 문제의 실재화 여부 판단용)
- **분야별 (선택 건수 / dedup 후 풀 크기) 비율**: 특정 분야가 지속적으로 MIN=8에 묶이는지 기록 (예: 경제·IT가 매일 8이면 ratio가 그 분야엔 무의미).

**임계는 데이터를 보고 정한다 (현재 임계 없음).** "주 N회 이상이면 대응" 같은 숫자를 지금 박으면 ADR-014가 비판받은 "ratio=0.15는 직관"과 똑같은 근거 없는 임계가 된다. 최소 1~2주 분포를 모은 뒤, 천장 발동 빈도의 실제 히스토그램을 보고 "이 빈도부터는 다양성 손실이 산출물에서 체감된다"는 지점을 잡아 임계를 설정한다. 즉 **임계 설정 자체가 보류 대상**이며, 그 전제는 위 두 지표의 운영 로그 축적이다.

**미해결 이슈**
- **MISC silent dropping 감지 (비판 수용 / ADR-012 후퇴 보완).** 분류 LLM이 메가 트렌드급 중요 사건을 MISC로 오분류하면 분석에서 통째로 누락되고, 산출물엔 안 나타나며 풀 통계엔 "MISC N건"으로만 묻힌다. ADR-005 시절엔 매체가 분야에 묶여 "정치 분량 비정상"이 즉시 보였으나, 분류 단계가 끼면서 누락이 분포 노이즈에 묻힌다(silent dropping). **저비용 1차 방어: MISC 비율 임계 경보** — MISC가 전체 풀의 일정 비율 초과 시 로그 경고. 의존성 0, 코드 몇 줄. 이 임계는 근거가 있다: 운영 2회 실측 MISC ≈ 10%라, 평시 10% 위에 마진을 둔 15%는 평시 오경보 없이 이상 급증만 잡는다(임의 숫자가 아니라 관측 분포 기반). 단 2회는 표본이 작아, 운영 누적 후 MISC 평시 분포가 확정되면 임계를 재조정할 수 있다. (분야 분포 표준편차 이탈 경보는 역사 데이터 누적 후, 배치 offset 일치율 회귀는 호출 비용 때문에 보류.)
- **MAX=20 천장에서 무엇이 잘리는지 통제 부재 (보류 — 측정 후 판단).** 본문 확보율 100% 상황에선 `select_for_analysis`의 본문 우선 정렬이 사실상 RSS 원본 순서로 붕괴해, 21번째 이후로 잘리는 기사가 임의에 가깝다. 같은 사건 클러스터가 상위 20건에 몰리면 다른 큰 사건이 통째로 빠진다. **단 현재까지 천장 발동 0회**(가장 큰 풀이 정치 dedup 후 7건). 급박 특보로 실제 발동해야 측정 가능 → 위 측정 계획의 천장 발동 빈도로 추적. 발동 시 잘린 기사의 클러스터 분포를 보고 인터리브 정렬 또는 2단 프롬프트 도입 판단.
- **MAX=20 천장에서 무엇이 잘리는지 통제 부재 (보류 — 측정 후 판단).** 본문 확보율 100% 상황에선 `select_for_analysis`의 본문 우선 정렬이 사실상 RSS 원본 순서로 붕괴해, 21번째 이후로 잘리는 기사가 임의에 가깝다. 같은 사건 클러스터가 상위 20건에 몰리면 다른 큰 사건이 통째로 빠진다. **단 현재까지 천장 발동 0회**(가장 큰 풀이 정치 dedup 후 7건). 급박 특보로 실제 발동해야 측정 가능 → 위 측정 계획의 "천장 발동 빈도"로 추적. 발동 시 잘린 기사의 클러스터 분포를 보고 인터리브 정렬 또는 2단 프롬프트 도입 판단.
- **Lost in the Middle (보류 — 측정 후 판단).** 입력이 길어질 때(MAX=20이 다수 분야서 동시 발동 시 80k+ 토큰 가능) Flash가 중간 기사를 무시하는 경향. 현재 실측 입력은 32k로 천장 미발동이라 잠재 위험. 측정 항목: 동일 입력 2회 호출 시 deep_issues 제목 집합 일치율(표본 점검, 상시 아님). 일치율이 낮으면 위치 편향 의심 → 분야 간 인터리브 정렬 검토.
- 같은 사건이 분야 분산되어 각 분야 dedup에서 따로 살아남는 문제 (AI 평가 갭 4)
- 분야 분류 정확도 보정 (경계 사례: "방송 접근권 TV"가 IT vs 사회)
- ~~인사이트 정형 패턴 ("향후 12개월 내 ~할 것") 고착화~~ **(프롬프트 대응, 2026-05-29 — 운영 검증 대기).** 3일 연속(05-25/26/27) 5개 분야 모두 "향후 12개월 내 ~"로 시작하는 정형 패턴 확정. 원인 진단: 직전 라운드에서 시스템 프롬프트에 "다음 12개월 내 발생 가능한 변화를 예측"이라고 적은 문구를 LLM이 그대로 베낌. 대응(analyzer SYSTEM_PROMPT): (1) "12개월" 등 고착된 시점 표현 삭제, (2) 3개 갈래(A 조건부 예측 / B 구조 신호 해석 / C 분야 간 파급) 중 선택하도록 구조 제시 (갈래 D 역사적 비교는 환각 위험으로 제외), (3) "5개 분야가 모두 같은 갈래 쓰면 안 됨" 메타 다양성 규칙, (4) 절대 금지 표현 명시 ("12개월 내", "ESG/디지털 전환/패러다임" 등 추상 키워드 끝맺음). 운영 1~2회로 검증. 직전 강화가 역효과 낸 전례가 있어 결과 보고 재조정 가능.

위 이슈는 운영 데이터 누적 후 처리. MISC 임계 경보는 저비용이라 우선 구현 후보.

**관련 결정**: [ADR-003 하이브리드 본문 선별](#adr-003-하이브리드-본문-선별), [ADR-012 LLM 사후 분류](#adr-012-rss-평탄화--llm-사후-분야-분류)

## ADR-015: 수집 윈도우를 cron 예정 시각 기준으로 고정

**Status**: Active

### 컨텍스트

수집 윈도우(어떤 발행 시각의 기사를 모을지)를 기존엔 **실행 시각(now)** 기준으로 계산했다(collectors.py: `start = now - 48h`, main.py: `date_str = now - 1일`). 문제:

- **재실행·지연 시 윈도우 평행이동.** 2026-06-01 브리핑이 503 장애로 실패 후 재실행됐는데, 첫 실행(08:27)과 재실행(13:25)의 윈도우가 5시간 어긋나 수집 290건 vs 277건, 분석 47건 vs 46건으로 갈렸다. 같은 날짜 브리핑인데 표본이 달라짐.
- **테스트 표본 비결정성.** 수동으로 두 번 돌리면 입력이 달라져, "어제 대비 분석 품질 변화"를 볼 때 입력 변동과 코드 효과가 뒤섞임.
- **대상날짜·윈도우 불일치 가능성.** main(대상날짜)과 collectors(윈도우)가 각각 `now`를 따로 읽어, 자정 근처 실행 시 둘이 어긋날 수 있었다.

기존 collectors 주석은 "수동 실행 시점에 상관없이 최근 기사 확보"를 48h 폭의 근거로 적었으나, 끝점이 `now`라 의도("시점 무관")와 구현("시점 연동")이 모순이었다.

### 결정

**cron 예정 시각(매일 07:00 KST)을 단일 기준점으로 삼아 대상날짜·윈도우를 모두 유도한다.**

- 공유 모듈 `timewindow.py`에 상수 `SCHEDULED_HOUR_KST = 7` + 계산 함수.
- 윈도우 = `[D-1 07:00, D+1 07:00)` (D = 대상날짜 = 어제). 끝점 = 예정 시각 = 오늘 아침.
- 폭 48h 유지 (RSS 반영 지연·매체 타임존 오차 보상). 끝점을 07:00으로 둬 **새벽 급보까지 포함** (사용자가 아침에 읽을 때 최신 반영).
- 상한(`pub >= end`)도 추가해 예정 시각 이후 기사는 배제(다음날 몫).

근거: GitHub Actions schedule은 예정 시각보다 **일찍 트리거되지 않고 지연만 발생**한다(공식 동작). 따라서 예정 시각을 윈도우 끝으로 박아도 미래 기사를 긁을 위험이 없고, 언제 실행하든(정시/지연/수동 재실행) 동일 표본이 보장된다.

### 결과

**장점**
- 재실행·Actions 지연·수동 실행에 무관하게 표본 고정 (테스트 비교 정확성 확보)
- 대상날짜·윈도우가 단일 기준점에서 유도되어 항상 정합 (자정 근처 불일치 제거)
- 48h 폭의 반영 지연 방어는 유지하면서 "시점 무관" 의도를 실제로 실현
- 회귀 테스트 `tests/test_timewindow.py`로 고정성·정합성 검증

**단점**
- `SCHEDULED_HOUR_KST` 상수와 `.github/workflows/daily.yml`의 cron(`0 22 * * *` = UTC 22:00 = KST 07:00)을 **수동 동기화**해야 함. 한쪽만 바꾸면 윈도우와 실제 실행이 어긋남. (테스트 `test_scheduled_hour_is_7`이 상수 변경을 감지하는 가드 역할)
- 48h 윈도우라 어제 브리핑에서 다룬 기사가 오늘 후보에 다시 오를 수 있음 (날짜 간 중복). 현재는 dedup이 같은 실행 내 중복만 제거하고, 날짜 간 중복은 미처리.

**미해결 이슈**
- **날짜 간 중복 기사 처리 (설계 원칙 확정, 구현 보류 — 측정 대기).** 48h 윈도우로 어제 분석한 기사가 오늘 또 후보에 오를 수 있다. 단 "중복"은 세 종류로 갈리고, 처리 방침이 다르다:

  | 유형 | 식별 | 처리 |
  |---|---|---|
  | (1) 완전 동일 기사 | 같은 article.link (URL) | **배제** — 글자 그대로 같은 기사, 어제 본 걸 또 볼 이유 없음 |
  | (2) 같은 사건 후속 기사 | 다른 URL, 새 국면 (예: 서소문 "사고 발생"→"운행 재개") | **보존** — 후속 보도는 새 정보 |
  | (3) 같은 사건 단순 반복 | 다른 URL, 새 정보 거의 없음 | **건드리지 않음** — LLM 사건 클러스터링에 맡김 |

  **설계 원칙: 배제는 article.link(URL) 완전 일치만.** URL이 다르면(2·3) 손대지 않는다. 이유: (2) 후속을 죽이면 안 되고, (3) 반복은 의미 판단이 필요해 LLM 영역이라 비싸고 불확실한데, 이미 사건 클러스터링이 5일 관찰상 잘 작동(중동 6건 묶기 등)하므로 추가 비용 불필요. URL 일치만으로 (1)은 100% 정확히 잡히고 (2)(3)은 자동 보존된다.

  **구현 전 선결 과제 (순서대로):**
  1. *측정*: 5일+ 백업/Notion에서 같은 URL 재등장 실제 빈도. 드물면(예: 5일에 1~2건) 이 작업 자체가 ROI 낮아 보류 유지. 잦으면 구현 착수.
  2. *상태 저장 위치*: Actions는 매 실행 초기화되므로 "어제 쓴 URL 집합"을 영속 저장해야 함. 1순위 후보는 **Notion 어제 페이지 역조회** (이미 출처 URL이 페이지에 있어 추가 저장소 불필요). 대안: 상태 파일 git 커밋(push 권한·이력 오염), gist(새 토큰). 
  3. *graceful 처리*: 어제 페이지 없음(첫 실행/실패일)·파싱 실패 시 배제 없이 정상 진행.

  배제 단위·보존 원칙은 위로 확정. 측정 결과 보고 구현 여부·상태 저장 방식만 결정하면 됨.

## ADR-016: Slack Incoming Webhook 알림 + Notion 블록 딥링크

**Status**: Active (2026-06-24)

### 컨텍스트

Notion에 브리핑이 올라가도 "오늘 뭐가 올라왔는지" 확인하려면 직접 앱을 열어야 한다. 솔로 운영(사용자 1인)에서 **푸시 알림**이 없으면 읽기 습관이 쉽게 깨진다.

선택지:
1. **Slack Incoming Webhook** — 앱 생성·서버 없이 HTTP POST 한 번으로 채널/DM에 전송. Block Kit으로 버튼·링크 지원. 무료 플랜에서도 동작.
2. Slack App (slash command, event subscription) — 인터랙션 엔드포인트(HTTPS 서버)가 필요. 솔로·서버리스 운영과 맞지 않음.
3. 이메일/SMS — 포매팅 제한, 딥링크 구현 복잡.
4. Notion 자체 알림 — DB 행 생성 시 알림 없음(공식 지원 아님).

### 결정

**Slack Incoming Webhook(옵션 1)**으로 구현. 추가 서버·인터랙션 엔드포인트 없이 webhookURL만 Secret으로 등록하면 동작.

**메시지 구조 (Block Kit)**:
- Header: 날짜
- TL;DR: 콜아웃 텍스트
- 분야별 섹션: 분야 이모지 + deep_issue 목록
- deep_issue 항목: 제목 + Notion 블록 딥링크 버튼 (`actions` block)
- 마지막에 Notion 전체 페이지 링크

**Notion 블록 딥링크 구성**:
- 페이지 업로드 후 `blocks.children.list` API로 전체 블록 조회
- 필터: `type == "heading_3"` AND `is_toggleable == True` AND 텍스트가 `"📊"` 시작
- URL = `https://www.notion.so/{page_id_no_hyphens}#{block_id_no_hyphens}`
- 조회 실패 시 빈 dict 반환 → Slack 메시지는 링크 없이 텍스트로 전송 (graceful degradation)

**두 기능을 별도 커밋으로 분리**:
1. 기본 Slack 알림 (webhook POST, 텍스트 메시지)
2. Notion 딥링크 연결 (블록 조회 + 버튼 URL 채우기)

이유: 딥링크는 API 추가 호출·URL 포맷 의존성이 있어 독립 기능으로 분리하면 롤백/디버깅이 용이.

**선택 안 한 것**:
- Slack App: 서버 필요. 솔로 운영에 오버엔지니어링.
- 구독자 목록: 수신자가 1인이라 DM Webhook으로 충분.

### 결과

**장점**
- Secret 1개(`SLACK_WEBHOOK_URL`) 추가만으로 활성화 (미설정 시 자동 스킵 — 기존 동작 유지)
- 오류(exit 2·3)도 원인과 함께 알림 → "오늘 실패 여부"를 Slack에서 확인 가능
- deep_issue 클릭 → Notion 해당 분석 블록으로 바로 이동 (독서 마찰 최소화)
- 기존 `requests` 의존성 재사용, 추가 패키지 없음

**단점**
- Slack Webhook URL은 1개 채널/DM에 고정 (채널 변경 시 새 URL 발급 필요)
- Notion 블록 조회 API 추가 호출 (실패 시 딥링크 없이 전송 — 치명적 아님)
- Notion 내부 URL 포맷이 변경되면 딥링크 작동 중단 (외부 의존)

**관련 파일**: `slack_writer.py` (신규), `notion_writer._fetch_issue_block_urls`, `main.py` step 7, `daily.yml` SLACK_WEBHOOK_URL Secret

## ADR-017: 분류 부분실패를 '미해결' 상태로 타입화 + 부분집합 회복

**Status**: Active (2026-06-29 구현 — Phase 0a·0b·1a·1b·2a·0c 완료. 단계별 작업·완료 기준은 PLAN.md 2026-06-29 추가 계획)

### 컨텍스트

적대적 재검토(2026-06-28~29) + 독립 서브에이전트 대안 탐색에서 분류 단계의 **silent error swallow**가 확인됐다.

분류는 100건 배치로 처리(ADR-013, `CLASSIFY_BATCH_SIZE=100`). `_classify_batch`는 LLM 호출/파싱 실패 시 예외 대신 **빈 리스트를 반환**한다(`classifier.py:241-249`). 배치 루프는 결과를 그대로 누적하므로(`classifier.py:300-306`), **일부 배치만 실패하면** 나머지가 비지 않아 "전량 실패 가드"(`classifier.py:309-312`)를 통과하고, 실패 배치 기사들은 `_apply_classifications`의 누락 폴백(`classifier.py:181-186`)에서 **`기타`(genuine MISC)로 강등**된다. 즉 "분류 실패"라는 정보가 정상 값으로 세탁되어 사라진다.

이게 스냅샷과 만나면 **박제**된다: `main.py:88-95`가 분류 후 **무조건** `save_snapshot`을 호출한다. 주석(`main.py:90-91`)은 "전량 폴백은 저장 안 함"이라 선언하지만 **구현이 없다**(주석만 있는 독립 정정 버그). 운영 ~300건=3배치에서 배치 1개 실패 ≈ 코퍼스 1/3이 분석에서 증발하고, 같은 날짜 재실행은 스냅샷을 재사용(분류 건너뜀, `main.py:86`)하므로 **회복 불가 — 재실행이 자가치유가 아니라 부분실패를 고착**시킨다. 전면 실패 시엔 전량 `기타` → `total=0` → 빈 페이지가 영구 박제된다.

`check_misc_ratio`(ADR-014 미해결 이슈로 도입)는 로그 전용이고 폴백-MISC와 genuine-MISC를 구분 못 해 이 실패를 가리지 못한다.

### 결정

**분류 결과를 3-상태로 타입화하고, 회복은 '미해결' 부분집합만 결정론적으로 겨냥한다.**

| 상태 | 의미 | 분석 | 재시도 |
|---|---|---|---|
| 5분야 라벨 | 정상 분류 | 포함 | X |
| `기타` (genuine MISC) | 성공 배치가 기타로 판정 | 제외 | **X (재굴림 금지)** |
| `미해결` (unresolved) | **배치가 알고리즘적으로 실패** | 잠정 제외 | **O (이것만)** |

- **판정은 LLM이 아니라 코드**: 배치가 `[]` 반환 → 그 배치 기사 전부 `미해결`. genuine MISC와 결정론적으로 구분.
- **2단 회복**:
  1. *런 내 재시도*: 실패 배치만 1~2회 재시도(Gemini 503은 대개 짧음). 잔여만 `미해결`로 남김.
  2. *런 간 회복*: 스냅샷에 `complete` + 기사별 `unresolved` 저장. `complete=False`면 재실행 시 **저장된 기사 위에서**(재수집 금지) `미해결`만 재분류 → 병합 → 해결 시 `complete=True`로 재저장.
- **Graceful(불가침)**: 회복 재실패 시 `미해결`→`기타` 폴백·무조건 진행. 스냅샷은 끝까지 "막지 않는 최적화"(snapshot.py 원칙).
- **결정론(ADR-015)**: 분류는 `temperature=0.5`(`llm.py:78`)라 비결정적 → **반드시 `미해결`만** 재분류한다(해결분을 재굴림하면 라벨이 바뀜). 인접 개선으로 분류 `temperature=0` 권장(라벨링 작업이라 타당, 회복 수렴 보장).

전면실패는 이 모델의 특수케이스로 흡수된다(전량 `미해결` → `complete=False` → 재실행 전량 재분류). 원안의 "전량이면 save 스킵" 가드는 불필요해진다 — 스킵은 재수집(표본 드리프트, ADR-015 위반)을 강제하지만, 완전성 저장은 표본을 보존하면서 회복한다.

### 대안

- **A. 부분실패 시 스냅샷 저장 보류 → 재실행 재수집·재분류.** 실격: 재수집이 RSS top-N 슬라이딩(314→127, snapshot.py 배경)을 재노출 → ADR-015 표본 고정 위반. 분류 버그를 표본 손실 버그로 맞바꾸는 셈.
- **C. 재실행 시 MISC 전량 재분류.** 기각: genuine MISC를 매 재실행 재굴림(temp 0.5) → 비결정·미수렴. 무료 티어 콜 낭비.
- **E. 배치 100→30 축소.** 기각: blast radius만 줄고 실패 수는 안 줆. 콜 3→9로 무료 티어 압박(ADR-013이 이미 기각).
- **F 단독(경보만).** 불충분: 가시화는 하나 박제를 못 막고 운영자 수동 삭제가 필요. 본 결정에 1b로 포함하되 단독으론 부족.
- **현행 유지(그냥 둔다).** 기각: 빈도는 낮아도 "실패를 정상값으로 세탁"은 빈도 무관한 에러 핸들링 결함이고, 전면실패 박제는 질적으로 다른 손상(빈 브리핑 영구화)이다.

### 결과

**장점**
- 실패가 타입으로 남아 관측·회복 가능(silent swallow 제거).
- 회복이 `미해결` 부분집합 한정 → 싸고(최대 1배치 1회) 수렴(`complete=True` 후 불변 — "매번 다시" 없음).
- 표본 보존(재수집 없음) → ADR-015 결정론 유지.
- 전면/부분 실패를 단일 모델로 처리.

**단점 / 트레이드오프**
- 2단 회복은 유일한 신규 경로 — "막지 않는 최적화" 컴포넌트에 새 실패 모드를 들인다. graceful 폴백으로 blast radius를 봉인.
- 스냅샷 스키마 변경(`_FIELDS` + `complete`) — 구버전 스냅샷 호환은 `load_snapshot` graceful(없는 필드 기본값) 처리.
- 분류 `temperature=0` 전환은 별도 검증 필요(분류 품질 회귀 없는지).

**미해결 / 후속**
- 회복 빈도·효과는 운영 로그로 관측(ADR-014/015 measure-first 기조).
- 단계별 작업·완료 기준은 PLAN.md(2026-06-29 추가 계획) 참조.

## ADR-018: robots.txt 준수 + 본문 크롤링을 선별 후보로 한정

**Status**: Active (2026-07-07 구현)

### 컨텍스트

공개 저장소 전환 점검(2026-07-07)에서 본문 수집의 법적 노출 지점 두 개가 확인됐다.

1. **robots.txt 미확인 크롤링**: RSS 피드는 매체가 배포 목적으로 제공하는 공식 채널이지만, 본문은 기사 **웹페이지**를 직접 요청해 추출한다(trafilatura/requests/bs4 3단계). 이 페이지 요청이 robots.txt를 확인하지 않았다. 실측(9개 매체 robots.txt + 실제 RSS 링크 판정 프로브) 결과 전 매체가 일반 UA(`User-agent: *`)에 기사 페이지를 허용하므로, 준수를 넣어도 본문 확보율 손실은 없다 — 차단은 주로 명명된 AI봇(ClaudeBot·GPTBot 등) 대상이고 우리 UA는 `*` 그룹을 따른다.
2. **전량 크롤링**: 수집 직후 윈도우 내 전 기사(~300건)의 페이지를 요청했지만, 분석에 실제 쓰이는 것은 분야별 선별 8~20건뿐. 불필요한 요청이 매체 서버 부담과 저작물 복제 범위(스냅샷→Actions 아티팩트에 본문 저장)를 키웠다.

### 결정

1. **robots.txt 게이트** (`collectors._robots_allowed`): 기사 페이지 요청(`fetch_body`, `_date_from_page`) 전에 호스트별 robots.txt 판정. 200→파싱, 4xx→허용(robots 부재 간주, RFC 9309), 5xx·네트워크 오류→보수적 차단. 호스트당 1회 캐시. 차단으로 본문을 못 얻어도 요약 폴백(`content_for_llm`)이 흡수하므로 파이프라인은 계속된다. RSS 피드 요청은 게이트 대상이 아니다(공식 배포 채널).
2. **본문 크롤링 지연·한정** (`enrich_with_bodies(target=...)`): 수집 직후 전량 크롤링을 제거하고, 분류·dedup 후 분야별 선별 후보에만 수행(main step 3). 최신 기사부터 시도해 본문 확보가 분야 목표(`target`, 8~20)에 닿으면 중단, 실패가 이어져도 시도 `target*3`에서 중단(요청 폭주 안전망).
3. **본문 반영 스냅샷 재저장**: 본문이 step 3에서 채워지므로 선별 후 스냅샷을 재저장 — 재실행이 같은 본문을 재사용해 재크롤링을 피하고 선별 결과도 1차 실행과 일치(ADR-015 결정론). ※ 같은 날 ADR-019가 스냅샷의 본문 저장 자체를 제거하면서 이 재저장은 삭제됨 — 재실행은 후보 재크롤링으로 복구.

### 함정 (실측)

`RobotFileParser.read()`를 쓰면 안 된다. urllib 기본 UA(`Python-urllib/3.x`)로 요청하는데 일부 매체(한겨레·매경·머니투데이)가 이를 403으로 거부하고, read()는 403을 **전체 차단**으로 해석한다 — 프로브에서 4개 매체가 오판으로 0% 허용이 됐다. 반드시 requests + `UA_DESKTOP`으로 받아 `parse()`에 넘긴다. 회귀 테스트: `tests/test_robots_gate.py`.

### 대안

- **정직한 봇 UA(예: NewsBriefingBot) 선언.** 보류: robots 판정은 어차피 `*` 그룹이라 동일하지만, 위 403 사례처럼 비브라우저 UA를 기계적으로 거부하는 매체가 있어 본문 확보율이 실측 불가 수준으로 흔들린다. robots 준수가 실질 보호를 제공하므로 UA 교체는 별도 판단.
- **robots 오류 시 허용(관대 판정).** 기각: "준수"의 목적이 법적 노출 축소인데 판단 불가를 허용으로 기울이면 취지가 약해진다. 본문 실패는 요약 폴백이 흡수하므로 보수 판정의 비용이 낮다.
- **선별 후 재크롤링 없이 요약만 사용.** 기각: 심층 분석 품질이 본문에 의존(하이브리드 선별의 존재 이유). 후보 한정 크롤링이면 요청 수가 이미 1/2~1/10로 준다.

### 결과

**장점**
- 매체 의사 표시(robots) 존중 — 법적·윤리적 노출 축소.
- 기사 페이지 요청 수 ~300건/일 → 분야별 목표 도달분(평시 ~60-100건)으로 감소.
- 스냅샷에 저장되는 본문(→ Actions 아티팩트 노출 면적)도 선별 후보분으로 축소. (이후 ADR-019가 본문 저장 자체를 제거)

**단점 / 트레이드오프**
- dedup 시점에 본문 유무를 모르므로(구버전은 본문 우선 보존) 유사 제목 중 어느 기사가 살아남는지가 달라질 수 있다 — 제목 유사 기사 간 차이라 분석 품질 영향은 미미.
- robots.txt가 5xx/네트워크 오류인 날은 해당 매체 본문이 전부 빠진다(요약 폴백) — 보수 판정의 의도된 비용.
- 재실행 시 미해결 회복으로 분야가 새로 배정된 기사는 본문 재크롤링이 필요할 수 있다(스냅샷 재저장이 반복 크롤링은 방지).

### 관련 결정

## ADR-019: 스냅샷에서 본문 제외 — 공개 아티팩트 저작권 노출 제거

**Status**: Active (2026-07-07 구현)

### 컨텍스트

스냅샷(ADR-015 표본 보존)은 backups/와 함께 Actions 아티팩트로 30일 보관된다. 공개 저장소에서는 GitHub 계정만 있으면 **누구나 아티팩트를 다운로드**할 수 있어, 스냅샷에 담긴 기사 본문 전문이 사실상 저작물 공중 전송이 된다 — 공개 전환 점검(2026-07-07)에서 최상위 법적 리스크로 확인. ADR-018이 크롤링·저장 범위를 선별 후보로 줄였지만, 저장 자체는 남아 있었다.

스냅샷의 가치를 분해하면 본문의 기여는 작다: ① 표본 보존(제목·링크·날짜·분야면 충분), ② 분류 LLM 호출 절약(분류는 제목 기반 — 본문 불필요), ③ 본문 재크롤링 회피·선별 결정론(본문 필요 — 그러나 ADR-018 이후 재크롤링은 분야별 후보 수십 건 수준으로 저렴).

### 결정

`snapshot._FIELDS`에서 `body`를 제거한다. 스냅샷 파일에 본문이 아예 기록되지 않으므로 아티팩트에 실리는 것은 제목·요약(RSS 제공분)·링크·분야뿐이다. 재실행(스냅샷 재개)은 main step 3에서 선별 후보만 재크롤링해 본문을 복구한다(ADR-018 경로 재사용 — 별도 코드 불필요). 구버전 스냅샷의 body 필드는 로드 시 무시된다(graceful). step 3의 본문 반영 재저장은 무의미해져 제거.

### 대안

- **B안: Notion 백업 DB로 이전(사용자 검토안).** 기각(보류): 비공개 저장이라 노출은 해소되고 본문 보존·결정론도 유지되지만, 블록 2,000자 제한(본문 분할 저장), 파이프라인 시작점의 새 실패 모드, 날짜별 페이지 정리 관리가 필요하다. ADR-018로 재크롤링이 싸진 뒤라 본문 보존을 위해 새 저장 인프라를 들일 비용 대비 이득이 작다. 재실행 결정론이 중요해지면 승격 후보.
- **아티팩트 암호화.** 기각: 공개 다운로드는 막지만 키 관리가 생기고, 본문 저장의 근본 필요가 약하다는 점은 그대로다.
- **아티팩트 업로드 자체 중단.** 기각: 스냅샷 복원(표본 보존)과 업로드 실패 시 로컬 백업 회수 경로가 사라진다 — 본문만 빼면 유지할 수 있는 가치.

### 결과

**장점**
- 본문이 러너 밖으로 나가는 경로가 LLM 호출(분석 목적 일시 이용)뿐 — 공개 전환의 최상위 리스크 소멸.
- 스냅샷 파일 크기 대폭 감소.
- 표본 보존·분류 LLM 절약은 그대로.

**단점 / 트레이드오프**
- 재실행 시 후보 재크롤링(분야별 목표 도달분) — 매체 요청 반복. ADR-018 상한(target*3)이 폭주를 막는다.
- 재크롤링 성공 여부가 1차 실행과 다르면 선별이 달라질 수 있음(결정론 약화). 재실행은 1차가 실패해 산출물이 없을 때만 일어나므로 실질 영향 제한적.
- 한겨레 prefetch 본문(_date_from_page 부산물)도 재개 시 소실 — 후보로 선별되면 재크롤링.

**후속**
- 공개 전환 전 **기존 아티팩트 삭제** 필요 — 과거 실행분 스냅샷에는 전량 본문이 남아 있다.
